# Release Notes

Please reference docs @ [dockview.dev](https://dockview.dev).

## 🚀 Features

- Touch and pointer-event drag-and-drop support. Internal drags (tabs, group headers, paneview headers, tab group chips) now work on touch devices via a pointer-event pipeline that runs alongside the existing HTML5 path.
  - **Touch users** get long-press initiation (250ms held, 8px tolerance) so quick swipes on the tab bar fall through to native horizontal scrolling. Once armed, a follow-finger ghost tracks the pointer and drop-zone overlays appear over targets.
  - **Mouse / pen on desktop** continues to use HTML5 DnD — preserves cross-window popout drags, OS file drops, iframe shielding, and `dataTransfer` for external receivers.
  - **Long-press context menus** on tabs and tab group chips bring right-click parity to touch.
  - **Close buttons on inactive tabs** are now visible on touch devices via `@media (hover: none)`. Hit area expanded to a finger-friendly size on coarse-pointer devices.
  - **Native horizontal scroll** on the tab bar via `touch-action: pan-x`.

## 🛠 Miscs

- The drop-target overlay rendering was extracted into a shared `dnd/dropOverlay.ts` module used by both the HTML5 `Droptarget` and the new `PointerDropTarget`, eliminating duplication of ~330 lines.
- Hit-test for nested drop targets now prefers the closest registered ancestor (per-tab over layout-root), fixing a regression where outer drop targets eclipsed inner ones.
- Iframe pointer-events shielding restored for both the HTML5 path (existing) and the pointer path (new).
- Popout-window pointer drags listen on the source's owning window (`element.ownerDocument.defaultView`) rather than the bare main `window`.
- `PointerGhost.update` uses `transform: translate3d(...)` instead of `style.left/top` so 60Hz pointermove updates don't trigger layout.
- `_findTargetUnder` maintains an incremental `Element → target` index keyed off `register()` / dispose, eliminating per-pointermove map allocations.

## 🔥 Breaking changes

- The `nativeEvent` field on the following events is widened from `DragEvent` to `DragEvent | PointerEvent`:
  - `DroptargetEvent`, `WillShowOverlayEvent`
  - `DockviewDidDropEvent`, `DockviewWillDropEvent`
  - `DockviewWillShowOverlayLocationEvent`
  - `TabDragEvent`, `GroupDragEvent`, `TabDropIndexEvent`
  - `DockviewDndOverlayEvent`, `PaneviewDndOverlayEvent`
  - The `event` parameter of `IDockviewGroupPanelModel.canDisplayOverlay` and the `CanDisplayOverlay` callback type
- Consumer code that reads `event.nativeEvent.dataTransfer` (e.g. to attach external-receiver payloads in `onWillDragPanel`/`onWillDragGroup`) must narrow first:
  ```ts
  api.onWillDragPanel((event) => {
      if (!(event.nativeEvent instanceof DragEvent)) {
          return; // touch drag — no DataTransfer
      }
      const dataTransfer = event.nativeEvent.dataTransfer;
      // ...
  });
  ```
  Touch (pointer) drags can't bridge to external HTML5 drop zones outside dockview anyway, so the narrow is a no-op for the use case the API was designed for. The `dnd-external` example has been updated.
